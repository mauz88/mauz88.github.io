Suricata IDS Test

This project demonstrates network intrusion detection using Suricata on an AWS EC2 instance. It captures alerts generated from a test Nmap scan and logs them in JSON and fast log formats.

View Project
  • Steps Performed:
    1. Installed Suricata on EC2 and prepared the ruleset (suricata.yaml).
    2. Ran Suricata in live capture mode on interface ens5.
    3. Executed an Nmap scan from a Windows machine targeting the EC2 instance.
    4. Collected and analyzed alerts in eve.json and fast.log.
  • Sample Alert (from eve.json): 
    {
  "timestamp": "2025-08-12T18:29:48.732867+0000",
  "flow_id": 1189257252974275,
  "in_iface": "ens5",
  "event_type": "alert",
  "src_ip": "85.255.233.2",
  "src_port": 47687,
  "dest_ip": "172.31.32.173",
  "dest_port": 22,
  "proto": "TCP"
}
  • Configuration: 
    # excerpt from suricata.yaml
var HOME_NET: "[192.168.1.0/24]"
af-packet:
  - interface: ens5
    threads: 2
    cluster-type: cluster_flow
    cluster-id: 99
rule-files:
  - scan.rules

SSH Brute Force Lab with Hydra

This lab demonstrates a controlled brute force attack simulation using Hydra against an AWS EC2 Linux instance running SSH. ⚠️ This was performed in a safe test environment with explicit authorization (my own AWS server).

View Project
  • Lab Setup:
    • Attacker: Windows laptop with WSL2 + Hydra installed
    • Target: AWS EC2 Linux (Amazon Linux 2) with SSH enabled
    • Authentication: password-based login (test user)
    • Dictionary: rockyou.txt
  • Steps Performed:
    1. Created an EC2 instance and added a testuser with a weak password.
    2. Restricted SSH access to my public IP via the AWS security group.
    3. Ran Hydra from WSL to brute force the SSH login.
    4. Captured logs from the target server to observe failed login attempts.
    5. Verified the successful brute force discovery.
  • Evidence:
    • Hydra output (discovered password)
    • SSH server logs (PAM attempts)
  • Lessons Learned:
    • Brute force attacks are noisy and leave traces in logs.
    • Use key-based authentication and disable password logins.
    • Add protections like fail2ban or rate limiting.

WPA2 handshake capture & offline password verification — lab exercise

This lab demonstrates an end-to-end workflow for capturing a WPA2 4-way handshake from a controlled access point and performing offline verification against that capture in a safe, isolated environment. The objective is to understand the wireless authentication process, recognise handshakes in packet captures, and evaluate password-strength assumptions without impacting third-party devices or networks.

View Project on GitHub
  • Disclaimer / Legal & ethical statement:

    This work was performed only on equipment I own or where I had explicit written permission to perform testing. Testing was conducted in an isolated lab environment with no production or third-party devices impacted. This repository is for educational and portfolio purposes only.

  • Objective:

    Demonstrate an end-to-end lab workflow for capturing a WPA2 4-way handshake from a controlled access point and performing offline password verification against the capture to illustrate how weak passphrases may be validated offline.

  • Hardware & software (summary):
    • Wireless adapter: Qualcomm Atheros AR9271 (lab device).
    • Operating system: Kali GNU/Linux 2025.3 (lab installation).
    • Tooling: aircrack-ng suite (used for capture/analysis). Exact versions are recorded in the repository.
  • Test network setup:
    • The target SSID and credentials were configured on a local test device acting as an access point with no Internet uplink.
    • Lab SSID lab1: Lab1
    • Lab password 123456780: (lab-only test credential)
    • All devices were dedicated to the lab and contained no personal or third-party data.
  • Methodology (high level):
    1. Prepared the wireless adapter for passive traffic collection in the isolated lab environment.
    2. Monitor mode
    3. Identified the target test AP and channel, then recorded packet captures until a WPA2 4-way handshake was observed.
    4. Target SSID
    5. Deauthenticate connected devices
    6. Captured WPA handshake
    7. Used offline verification methods to test candidate passphrases against the captured handshake (tooling and wordlists referenced in the repository).
    8. Decode the PSK
    9. Documented results and restored all test hardware and network interfaces to normal managed mode after testing.
  • Results:
    • A valid WPA2 4-way handshake was captured for the lab AP.
    • The controlled lab password was validated against the capture using an offline wordlist (demonstrating the importance of passphrase strength).
    • All devices and interfaces were returned to their original state after testing.
  • Evidence & artifacts:

    Non-sensitive artifacts are available in the repository. Sensitive fields (e.g., full MAC addresses) have been redacted in public assets. Where applicable, SHA256 hashes for capture files are included so reviewers can verify integrity.

    • Repository (sanitised captures & metadata)
  • Lessons learned & mitigations:
    • Weak passphrases can be verified offline if an attacker obtains a handshake. Use long, random passphrases or passphrases with high entropy.
    • Consider using WPA3 or enterprise authentication (802.1X) where possible.
    • Monitor for anomalous wireless events and implement logging/alerting to detect potential reconnaissance.
  • Responsible disclosure & ethics:

    If you discover security issues on third-party networks, obtain explicit permission and follow a responsible disclosure process. This project adhered to a strict "lab only" policy.
